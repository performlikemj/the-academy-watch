# iOS app shell ‚Äî Mac-side runbook

The Academy Watch's iOS app is a **Capacitor wrapper around the existing React/Vite
SPA** (`academy-watch-frontend/`) ‚Äî not a rewrite. `dist/` gets copied into a native
Xcode project (`academy-watch-frontend/ios/`) and runs inside a WKWebView. See
`ios-migration-brief.md` (Phase 1 brief) for the product/architecture decision and
`docs/agents/frontend.md`'s "Native iOS is coming" note for the web-side conventions
this depends on.

This doc is written for MJ, on a Mac. Everything below this point requires Xcode ‚Äî
none of it can be done from Linux/CI.

## What's already done (Linux-side, this pass)

- `@capacitor/core`, `@capacitor/app`, `@capacitor/share` added as dependencies;
  `@capacitor/cli`, `@capacitor/ios` as devDependencies. `pnpm-lock.yaml` updated.
- `academy-watch-frontend/capacitor.config.json` ‚Äî see field reference below.
- `src/lib/platform.js` ‚Äî `isNativeApp()` + `useDeepLinks()` (contract in the brief).
- `src/lib/share.js` ‚Äî `nativeShare()`/`canNativeShare()` now use `@capacitor/share`
  inside the native shell, falling back to the Web Share API on web.
- `academy-watch-frontend/ios/` ‚Äî **the native Xcode project, generated by
  `pnpm exec cap add ios` and committed.** `cap add ios` worked fine on Linux (it's
  just scaffolding files, no build), so this step does not need to be repeated on a
  Mac. Only regenerate it if the folder goes missing or you intentionally want a
  clean scaffold (`rm -rf ios && pnpm run build && pnpm exec cap add ios`).
- `public/.well-known/apple-app-site-association` ‚Äî inert placeholder, see
  "Universal links" below.
- `public/staticwebapp.config.json` ‚Äî serves the AASA file as `application/json` with
  no SPA rewrite.
- `academy-watch-backend/src/main.py` ‚Äî a comment near the CORS block pointing back
  here; no code change (CORS is env-config only).

## Prerequisites

- **macOS** with **Xcode** (current stable ‚Äî Xcode 16.x as of this writing; check
  Capacitor 8's compatibility notes if a newer major Xcode is out).
- **Node 20** + **pnpm 10** (`corepack enable` will pick up the `packageManager` field
  in `package.json` automatically).
- An **Apple Developer Program** account (for a real Team ID, device provisioning,
  and App Store Connect).
- **CocoaPods is NOT required for this project today.** Capacitor 8 wires plugin
  dependencies through Swift Package Manager (`ios/App/CapApp-SPM/`), not
  `Podfile`/CocoaPods ‚Äî there's no `Podfile` in the generated project. If a future
  plugin only ships a CocoaPods integration, `cap sync ios` will generate a `Podfile`
  automatically and you'll need `sudo gem install cocoapods` at that point, but as of
  `@capacitor/core@8.4.1` / `@capacitor/app@8.1.0` / `@capacitor/share@8.0.1` it's
  pure SPM.

## capacitor.config.json field reference

JSON can't carry inline comments, so the field-by-field explanation lives here
instead (a `.ts` config was considered per the brief, but `@capacitor/cli` requires a
`typescript` devDependency to load `.ts` configs and this is otherwise a plain-JS
project with no `tsconfig.json` ‚Äî not worth the extra dependency for four fields):

| Field | Value | Meaning |
|---|---|---|
| `appId` | `com.theacademywatch.app` | Reverse-DNS bundle identifier. **PLACEHOLDER.** Must match the bundle ID registered in Apple Developer / App Store Connect before the first TestFlight/App Store submission. Changing it *after* first submission requires a brand-new App Store Connect app record, so lock it in during one-time setup below, not later. |
| `appName` | `The Academy Watch` | Display name under the home-screen icon and in system UI (share sheet attribution, permission prompts). |
| `webDir` | `dist` | Directory Capacitor copies into the native project on `cap sync`. Matches Vite's default build output (`vite.config.js` doesn't override `build.outDir`). |
| `ios` | `{}` | No overrides ‚Äî safe-area handling (tab bar, fixed elements) is done in CSS via `env(safe-area-inset-*)` by the shell agent's work, not native config. No `server.url` block either: the app ships the built `dist/` bundle, it does not point the WKWebView at a live remote URL. |

## One-time setup (per machine / per first release)

1. `cd academy-watch-frontend && pnpm install` (uses the committed lockfile).
2. Confirm `ios/` is present (it's committed ‚Äî see above). If missing:
   ```bash
   VITE_API_BASE=https://<real-backend-fqdn>/api pnpm run build
   pnpm exec cap add ios
   ```
3. Open the Xcode project:
   ```bash
   open ios/App/App.xcodeproj
   ```
4. **Bundle ID + signing**: select the `App` target ‚Üí *Signing & Capabilities*.
   - Set your **Team** (Apple Developer Program membership).
   - Replace the placeholder **Bundle Identifier** (`com.theacademywatch.app`) with
     the real one you register in App Store Connect ‚Äî and update
     `capacitor.config.json`'s `appId` to match, then re-run `pnpm run sync:ios` so
     the two never drift.
5. Register the app in [App Store Connect](https://appstoreconnect.apple.com) with
   that bundle ID (needed before the first TestFlight upload).
6. **Associated Domains** (for universal links ‚Äî see below): *Signing &
   Capabilities* ‚Üí **+ Capability** ‚Üí **Associated Domains** ‚Üí add
   `applinks:theacademywatch.com` (swap in the real production domain).
7. **App icon / launch screen**: `ios/App/App/Assets.xcassets/AppIcon.appiconset` and
   `LaunchScreen.storyboard` are Capacitor's default placeholders ‚Äî not scaffolded
   with real art in this pass. Fill in before the first submission (Apple rejects
   apps with the default icon).

## Every-release flow

1. Get the real backend FQDN the same way CI does (see
   `.github/workflows/deploy.yml` / `deploy-frontend.yml`):
   ```bash
   az containerapp show \
     -g rg-loan-army-westus2 -n ca-loan-army-backend \
     --query properties.configuration.ingress.fqdn -o tsv
   ```
2. Build with that as `VITE_API_BASE` (the `build:app` script fails loudly if you
   forget it ‚Äî see `academy-watch-frontend/package.json`):
   ```bash
   VITE_API_BASE="https://<fqdn>/api" pnpm run build:app
   ```
3. Sync the built web assets + native plugins into the Xcode project:
   ```bash
   pnpm run sync:ios
   ```
4. `open ios/App/App.xcodeproj`, bump the version (`MARKETING_VERSION` /
   `CURRENT_PROJECT_VERSION` build settings ‚Äî currently `1.0` / `1`).
5. *Product ‚Üí Archive*, then use the Organizer window to distribute to App Store
   Connect / TestFlight.

## Universal links

Two mechanisms, both handled by `useDeepLinks()` in `src/lib/platform.js`:

- **Universal links** (`https://theacademywatch.com/players/123` opens the app
  directly, and still works as a normal web link if the app isn't installed). This
  needs:
  - `public/.well-known/apple-app-site-association` ‚Äî currently a placeholder:
    `"appID": "TEAMID.com.theacademywatch.app"`. **Replace `TEAMID`** with your real
    10-character Apple Developer Team ID (Xcode ‚Üí Account ‚Üí Membership Details, or
    `App Information` in App Store Connect) in both the `appID` and `appIDs` entries.
  - The Associated Domains entitlement from one-time setup step 6, pointing at the
    real production domain.
  - The file must be served over **https**, with **no redirect**, at exactly
    `/.well-known/apple-app-site-association`, with `Content-Type: application/json`
    (extensionless files aren't auto-typed by Azure Static Web Apps otherwise).
    `public/staticwebapp.config.json` already has a route rule forcing that
    content-type and excludes `/.well-known/*` from the SPA `navigationFallback`
    rewrite, so it isn't swallowed by the `index.html` catch-all. Verify after
    deploy:
    ```bash
    curl -sI https://theacademywatch.com/.well-known/apple-app-site-association
    # expect: HTTP/2 200, content-type: application/json, no Location header
    ```
- **Custom URL scheme** (`com.theacademywatch.app://players/123`) ‚Äî registered in
  `ios/App/App/Info.plist` (`CFBundleURLTypes`). Needs no signing/Team ID, so it's
  the fastest way to test deep links on a simulator before Associated Domains is
  wired up:
  ```bash
  xcrun simctl openurl booted "com.theacademywatch.app://players/123"
  ```

`useDeepLinks()` parses either form down to an in-app path (e.g. `/players/123`) and
navigates the router there; see the function-level comments in `platform.js` for how
the two URL shapes are normalized (WHATWG URL parsing treats a custom scheme's first
path segment as `host`, not `pathname` ‚Äî the code corrects for that).

## CORS ‚Äî the operator step

`academy-watch-backend` reads `CORS_ALLOW_ORIGINS` from the environment
(`src/main.py`, near the top). If it's **unset**, the backend already allows `*`, so
the native app talks to it with no changes needed ‚Äî this covers dev/testing today.

The native shell's outgoing request `Origin` header is `capacitor://localhost` (and
`ionic://localhost` on older Capacitor/Cordova setups) ‚Äî **not** a real https origin,
because requests come from the WKWebView, not a hosted page. Once
`CORS_ALLOW_ORIGINS` is ever set explicitly on the backend Container App (moving off
the `*` default ‚Äî e.g. when tightening CORS for the web app), you must append both of
those to the comma-separated allowlist or the native app's API calls will start
failing CORS preflight:

```bash
# Check the current value first:
az containerapp show -g rg-loan-army-westus2 -n ca-loan-army-backend \
  --query "properties.template.containers[0].env[?name=='CORS_ALLOW_ORIGINS'].value" -o tsv

# Then set it, including the existing web origin(s) plus the two native ones:
az containerapp update -g rg-loan-army-westus2 -n ca-loan-army-backend \
  --set-env-vars CORS_ALLOW_ORIGINS="https://<swa-frontend-domain>,capacitor://localhost,ionic://localhost"
```

Do not hardcode these in `main.py` ‚Äî this is an env-only operator step by design (see
the comment left in `main.py` near the CORS block).

## App Store review cautions already handled by the shell

Per Apple guideline 3.1.1 (no external purchase links) and MJ's product call that
admin has no place in the consumer app:

- All `/admin` routes and admin nav entries are hidden when `isNativeApp()` is true
  (implemented in `App.jsx` by the shell agent).
- The Buy-Me-a-Coffee button and `/pricing` page/nav entry are hidden in the native
  shell for the same reason (implemented by the engagement agent).

Both gate on `src/lib/platform.js`'s `isNativeApp()`, which is `false` everywhere on
web (including mobile web) ‚Äî nothing is hidden from browser users, only from the
native binary Apple reviews. Re-verify these are still wired up before every App
Store submission; a regression here is a rejection, not just a bug.
